# “扫描录入病例”功能深度研究与移动端网站落地方案

## 功能定位与总体判断

基于你们这个面向独居长者的医疗建档与照护接续网站，我建议把新功能定义为“病例扫描录入”，目标不是“自动诊断”，而是把纸质病历、出院小结、处方、检验单等资料，通过手机网页转成“原件留存 + OCR 文本 + 结构化字段 + 人工确认后入档”的一体化流程。就 Web 能力而言，移动网页完全可以同时支持“从相册上传图片”和“直接调用摄像头拍照”，但实际产品设计不能只押注单一路径：`capture` 方案在浏览器支持上有限，而 `getUserMedia()` 适合做实时取景、重拍、裁边与扫描增强，但要求 HTTPS 安全上下文。citeturn2search1turn0search0turn19search0

从产品和技术两端综合看，最稳妥的实现方式是“移动端 H5 采集 + 服务端 OCR/文档解析 + 低置信字段人工校对 + 结构化归档”。原因很直接：浏览器内 OCR 虽然可做，但主流方案 Tesseract.js 官方就明确说明其核心目标是把 Tesseract 带到浏览器和 Node.js，且它本身不支持 PDF，也不改进底层识别模型；相比之下，PaddleOCR 的 PP-Structure 已面向图片/PDF 文档解析、版面分析、表格识别和 Markdown 还原，而 Google Cloud Vision、Azure Document Intelligence 也都提供密集文档 OCR、手写识别、布局分析与层级结构输出，更适合病例类材料。citeturn9search0turn9search1turn6search0turn6search2turn6search5turn4search0turn4search2turn5search1turn5search7

如果你们后续希望和医院、护理站、社工机构或其他健康档案系统对接，这个模块最好从第一天起就采用“原始文档”和“结构化健康信息”分层存储。HL7 FHIR 的 `DocumentReference` 适合保存扫描件及其元数据，而诊断、用药、过敏和检验/生命体征等信息分别可以映射为 `Condition`、`MedicationStatement`、`AllergyIntolerance`、`Observation` 等资源；即使首版不直接暴露 FHIR API，内部字段设计先向这套模型靠拢，后续扩展会轻松很多。citeturn11search1turn13search3turn11search2turn12search1turn12search0

## 手机网页采集方案

在前端交互上，建议做成双入口而不是单按钮。页面上同时放“拍照扫描”和“从相册上传”两个入口；“拍照扫描”优先尝试后置摄像头实时取景，“从相册上传”永远保留，作为所有机型的兜底路径。这样做的依据是：`capture` 属性确实可以提示系统优先使用设备相机，而且常见值包括 `user` 与 `environment`，但 MDN 明确标注它“Limited availability”，说明不能把它当作唯一方案；相反，`getUserMedia()` 可以请求实时视频流，再通过 `facingMode: "environment"` 或 `facingMode: { ideal: "environment" }` 优先拿到后置摄像头。citeturn2search1turn19search0turn0search0

```html
<meta name="viewport" content="width=device-width, initial-scale=1" />

<input
  id="scanUpload"
  type="file"
  accept="image/*"
  capture="environment"
/>

<video id="cameraPreview" playsinline></video>
```

```js
const video = document.getElementById("cameraPreview");

async function openRearCamera() {
  const stream = await navigator.mediaDevices.getUserMedia({
    video: { facingMode: { ideal: "environment" } },
    audio: false,
  });
  video.srcObject = stream;
}
```

上面这个组合就是首版最实用的 H5 基础形态：`<input type="file" accept="image/*" capture="environment">` 负责“轻量拍照/上传”，`getUserMedia()` 负责“完整扫描模式”。如果你们要做实时取景页面，`<video>` 建议显式加 `playsinline`，因为 WebKit 已明确说明在 iPhone 上，带 `playsinline` 的视频元素可以行内播放，不会自动全屏；而没有它的 `<video>` 仍可能触发全屏行为。citeturn2search1turn3search0

网站本身必须按 mobile-first 做，而不是桌面页面缩小。web.dev 的响应式设计指南明确建议在页面头部设置 `viewport`，用灵活布局和媒体查询适配不同宽度，并避免内容超出视口导致横向滚动；同时，无障碍响应式设计还特别强调不要通过 `maximum-scale=1` 或 `user-scalable=no` 禁用用户缩放，因为这会直接伤害需要放大阅读的用户。对长者场景，这一点非常重要。citeturn1search0turn1search6

扫描体验本身不要停留在“上传一张图”这么浅的层级，至少要有四个前置处理动作：自动裁边、透视矫正、亮度/对比度增强、重拍提醒。OpenCV.js 官方教程和 OpenCV 的 JS 文档都展示了文档扫描中常见的透视变换与阈值化处理，尤其是自适应阈值在光照不均的纸质材料上更适合 OCR；而 Tesseract 官方文档也明确提醒，为了得到更好的 OCR 结果，往往需要先改善输入图像质量。换句话说，前端做基础扫描增强，不是“锦上添花”，而是病例识别准确率的第一道门槛。citeturn10search0turn10search3turn10search4turn7search0turn7search1

## OCR 与结构化建档流程

我不建议把“纯浏览器 OCR”作为核心生产链路。Tesseract.js 的优势是完全可以运行在浏览器里，支持多语言、浏览器和 Node.js 环境，并适合做轻量离线或隐私优先的演示型功能；但官方 README 也写得很清楚，它不支持 PDF，而且不负责改进底层识别模型。对于你们要处理的病例、检验单、处方、出院小结这类“版式复杂、字体不统一、可能多页、可能包含表格”的材料，它更适合作为局部增强或离线兜底，不适合作为主引擎。citeturn9search0turn9search1turn8search4

更适合作为主链路的是服务端文档 OCR。若你们希望自建、可控、尽量不把健康数据交给第三方，PaddleOCR 是更贴近需求的第一选择，因为其文档解析链路已经覆盖图片/PDF、布局分析、表格识别、阅读顺序恢复和 Markdown 输出；如果你们后面要处理更多手写病例、拍照角度复杂、或者需要更强的现成文档解析能力，也可以评估云服务：Google Cloud Vision 的 `DOCUMENT_TEXT_DETECTION` 明确针对密集文档文本，且可用于手写文本提取；Azure Document Intelligence 的 Read/Layout 模型则支持更高分辨率的文档 OCR、手写/印刷文本、段落、单词、表格和结构分析。citeturn6search0turn6search2turn6search5turn4search0turn4search2turn5search1turn5search5turn5search7

在字段抽取上，推荐采用“文档分类 → OCR → 字段抽取 → 置信度判断 → 人工校对 → 入档”的顺序，而不是 OCR 完直接写库。原因在于主流文档智能系统本身就能返回层级化结构和字段/词级置信度：Google Vision 会返回 page/block/paragraph/word 等层级，Azure Document Intelligence 会返回字段和词级置信度。基于这些能力，最省人力的做法不是让志愿者整页重录，而是只把低置信字段、高影响字段和格式异常字段拉出来校对，例如“诊断名称、出院日期、药品名、用法用量、过敏史、关键检验值”。citeturn4search0turn4search3turn18search2

从落地优先级看，我的建议很明确：首版就把“文档类型识别”和“字段抽取”做成半自动。用户上传后，系统先判断它更像“出院小结”“处方”“检验报告”还是“门诊病历”，然后呈现对应字段模板。这样既能降低人工编辑成本，也能避免把所有病例都塞进同一张通用表单，导致后续查询和照护交接混乱。这里不一定要一步到位做特别复杂的 AI；首版用规则 + OCR + 少量模型即可，关键是把“上传即入档”和“人工确认后入档”区分开。citeturn6search0turn6search5turn18search2

## 数据模型与系统架构

在数据结构上，建议你们始终保留两层内容。第一层是“原始材料层”，也就是拍照后的 JPEG/PNG 或后续合成的 PDF；第二层是“结构化字段层”，也就是从病例中抽取出的诊断、用药、过敏、关键观察值、随访建议等。FHIR 的 `DocumentReference` 就是描述文档本身和元数据的天然对象，而 `Condition`、`MedicationStatement`、`AllergyIntolerance`、`Observation` 可以分别承载诊断、用药、过敏和检验/生命体征事实。这样做的最大价值不是“为了标准而标准”，而是未来你们做搜索、统计、照护交接、提醒、外部接口时，不必重新拆文档。citeturn11search1turn13search3turn11search2turn12search1turn12search0

一个适合你们的网站架构，可以理解为下面这条链路：

```text
手机网页
  ├─ 从相册上传
  └─ 调用摄像头实时扫描
          ↓
图像预处理
  ├─ 自动裁边
  ├─ 透视矫正
  ├─ 压缩与去噪
  └─ 质量检查
          ↓
OCR / 文档解析服务
  ├─ 文档类型判断
  ├─ 文本识别
  ├─ 表格/版面解析
  └─ 字段抽取与置信度输出
          ↓
人工校对台
  ├─ 只校对低置信或关键字段
  └─ 确认后保存
          ↓
居民档案
  ├─ 原始扫描件
  ├─ OCR 文本
  ├─ 结构化病例字段
  └─ 照护接续记录
```

如果你们当前还没有成熟的临床数据标准团队，也不要被 FHIR 吓住。首版完全可以先做一个“FHIR 思想对齐”的内部模型，比如文档表、诊断表、药品表、过敏表、观察表，字段命名尽量贴近标准的最小语义单元。这样，前期开发速度不会被标准实现拖慢，但后续如果要做 API、对接第三方系统、做可交换健康档案，也不会推倒重来。citeturn11search1turn13search4turn12search0

## 合规、安全与老年友好要求

这个功能一旦上线，就会直接处理“医疗健康信息”，而这在中国《个人信息保护法》中属于敏感个人信息。法律文本明确把“医疗健康”列入敏感个人信息类别，并要求处理此类信息必须具有特定目的、充分必要性并采取严格保护措施；处理敏感个人信息应取得单独同意，还应额外告知处理该类信息的必要性和对个人权益的影响。也就是说，你们不能只在注册页放一份笼统用户协议，而应该在“扫描录入病例”前单独弹出授权说明，写清楚采集目的、使用范围、保存期限、是否共享、如何删除与撤回。citeturn14search1turn14search2

此外，按照《个人信息保护法》第五十五条，处理敏感个人信息前应进行个人信息保护影响评估，并保留评估报告和处理记录；一旦发生泄露、篡改、丢失等事件，还需要立即采取补救并按法定要求通知。对你们这个项目来说，这意味着在上线前就要形成一份最小可用的 PIPIA 文档，至少覆盖数据流、处理目的、风险点、权限控制、日志留痕、存储位置、删除策略和应急响应。citeturn15search0turn15search1turn15search2

上传安全方面，别把它当成普通图片上传。OWASP 的文件上传清单明确建议采用白名单扩展名、不要信任客户端传来的 Content-Type、对文件签名做校验、使用随机文件名、设置大小限制、尽量将文件存储在 webroot 之外或独立服务器上，并在条件允许时增加杀毒或沙箱检查。对病例扫描功能来说，最少也要做到：只允许图片和必要文档格式、服务端二次校验、统一重命名、限制单文件大小、原图隔离存储、下载走鉴权接口而不是直接暴露公开 URL。citeturn16search0turn16search1

长者友好设计同样不能省。响应式方面要保证手机默认单列布局，避免横向滚动；可访问性方面要允许浏览器缩放，不要关掉 zoom；交互控件建议按 web.dev 的移动无障碍建议做到约 48×48 设备无关像素，至少满足 WCAG 2.2 对 24×24 CSS 像素或足够间距的最低要求。对“拍照扫描、重拍、确认保存、删除页面”这些关键按钮，宁可做大，也不要为了“界面精致”做小。citeturn1search0turn1search6turn17search0turn17search1

## 推荐落地方案

如果以“尽快可用、适合公益项目、兼顾合规与后续扩展”为目标，我建议你们按下面三层推进。

- **首版可用层**：手机网页支持相册上传和后置摄像头拍照，前端完成基础裁边/压缩，后端完成 OCR、字段抽取和人工确认后入档。这一版先把“原件保存 + 可搜索 + 可校对”做扎实。`capture` 负责轻路径，`getUserMedia()` 负责完整扫描路径。citeturn2search1turn0search0turn19search0
- **增强效率层**：加入自动透视矫正、文档分类、关键字段模板、低置信重点校对，把人工从“整页录入”降为“关键字段确认”。图像增强和结构化解析能力可分别由 OpenCV.js 与 PaddleOCR/云文档智能承担。citeturn10search0turn10search4turn6search0turn6search2turn5search1turn18search2
- **外出采集层**：若后续有志愿者上门采集、社区场景弱网明显，可以把站点进一步做成可安装 PWA。web.dev 明确说明安装体验依赖 Web App Manifest，而离线支持需要 Service Worker/Workbox 明确配置，并不会因为“是 PWA”就自动生效。citeturn20search0turn20search1turn20search2

如果要把这段直接写进你们的方案书，我建议可以用下面这段产品定义：

> 新增“病例扫描录入”模块，支持用户通过手机网页从相册上传病例图片，或直接调用后置摄像头进行拍照扫描；系统对图片进行自动裁边、透视矫正和清晰度优化后，完成 OCR 识别与字段抽取，并以“原始扫描件 + 结构化病例信息 + 人工校对确认”的方式归入长者个人档案，用于后续医疗医愿建档、照护交接、随访提醒与服务协同。

最终建议可以压缩成一句话：**做一个 mobile-first 的响应式 H5 网站，前端负责“采集和扫描增强”，后端负责“OCR 和结构化建档”，中间必须有“低置信字段人工确认”，底层同时保留“原始病例图像”和“结构化健康信息”两层数据。** 这样最符合你们当前项目的公益属性、移动端使用场景和后续可扩展性。
